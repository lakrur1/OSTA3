Порівняльний аналіз розробки

Виконавець: Єгор Крупєй

Мета роботи

Проведення порівняльного аналізу десктоп-версії (Electron) та веб-версії (REST API) системи управління файлами. Виділення спільних елементів, відмінностей, переваг та недоліків кожного підходу. Обґрунтування оптимальної технології.

2. Спільні елементи

2.1. Backend архітектура

  • Однаковий Django backend з REST API;
  • PostgreSQL база даних;
  • JWT токени для аутентифікації;
  • Ідентичні models (User, FileMetadata);
  • Однакові endpoints та бізнес-логіка.

2.2. Frontend код

  • 80% коду спільний (auth.js, fileList.js, fileViewer.js, api.js);
  • Однакова логіка роботи з API;
  • Ідентичний UI/UX дизайн;
  • Спільні CSS стилі.

2.3. Функціонал

  • Аутентифікація та авторизація;
  • Завантаження, перегляд, редагування, видалення файлів;
  • Сортування за назвою (варіант 06);
  • Фільтрація за типами .cpp та .png (варіант 06);
  • Перегляд .txt та .jpg файлів;
  • Drag-and-drop завантаження;
  • Спільний робочий простір.

3. Відмінності

3.1. Архітектурні

Десктоп:

  • Electron main process + renderer process;
  • Node.js integration через preload bridge;
  • Локальне зберігання даних можливе;
  • Прямий доступ до OS APIs.

Веб:

  • Single page application;
  • Sandbox браузера;
  • Всі дані на сервері;
  • Обмеження браузерних APIs.

3.2. Функціональні

Десктоп:

  • Синхронізація локальних папок з сервером;
  • Запис файлів напряму на диск;
  • Читання структури директорій;
  • Можливість offline кешування.

Веб:

  • Лише manual завантаження/вивантаження;
  • Завантаження через browser download;
  • Обмежений File API;
  • Завжди online.

3.3. Deployment

Десктоп:

  • Збірка окремих інсталяторів для кожної OS;
  • Розповсюдження через сайт або магазини;
  • Version management складніший;
  • Electron builder для пакування.

Веб:

  • Один deployment на сервері;
  • Доступ через URL;
  • Простіше оновлювати;
  • Django dev server.

4. Переваги та недоліки

4.1. Десктоп-версія (Electron)

Переваги:

  • + Повний доступ до файлової системи;
  • + Синхронізація папок автоматична;
  • + Краща продуктивність для heavy операцій;
  • + Можливість offline роботи;
  • + Native look and feel;
  • + Більше контролю над середовищем;
  • + Безпосередній доступ до OS APIs.

Недоліки:

  • - Великий розмір (~150-200 MB);
  • - Складніша розробка (main + renderer);
  • - Потребує інсталяції;
  • - Оновлення вимагають дій користувача;
  • - Підтримка різних OS версій;
  • - Більша поверхня атаки (native access);
  • - Вища вартість розробки та підтримки.

4.2. Веб-версія (REST API)

Переваги:

  • + Універсальний доступ з будь-якого пристрою;
  • + Малий розмір (2-5 MB);
  • + Не потребує інсталяції;
  • + Автоматичні оновлення;
  • + Простіша розробка та підтримка;
  • + Sandbox безпека браузера;
  • + Краща масштабованість;
  • + Нижча вартість.

Недоліки:

  • - Обмежений доступ до файлової системи;
  • - Немає повноцінної синхронізації;
  • - Залежність від інтернету;
  • - Обмеження браузерних APIs;
  • - Менше контролю над середовищем;
  • - Може бути повільніше на слабких з'єднаннях.

5. Обґрунтування оптимальної технології

5.1. Гібридний підхід (рекомендація)

Оптимальне рішення – підтримувати обидві версії:

Веб-версія як основна:

  • Для більшості користувачів достатньо базового функціоналу;
  • 80-90% use cases покриваються веб-версією;
  • Нижча вартість підтримки;
  • Більша аудиторія (доступ з мобільних).

Десктоп-версія як premium:

  • Для power users що потребують синхронізації;
  • Для користувачів з поганим інтернетом;
  • Для корпоративних клієнтів з особливими вимогами.

Переваги гібриду:

  • Спільний backend (80% коду);
  • Спільний frontend core (70% коду);
  • Максимальне покриття аудиторії;
  • Гнучкість у виборі для користувача.

5.2. Якщо вибирати одну технологію

Для даного проекту (file manager) – ДЕСКТОП

Обґрунтування:

  1. Основна цінність – робота з файлами та синхронізація;
  2. Веб-версія не може повноцінно реалізувати синхронізацію папок;
  3. Користувачі file manager очікують native функціонал;
  4. Конкуренти (Dropbox, Google Drive) мають десктоп клієнти.

Винятки (коли обрати веб):

  • Бюджет обмежений;
  • Потрібен швидкий launch;
  • Пріоритет – максимальна аудиторія;
  • Синхронізація не критична.

5.3. Майбутні покращення

Для обох версій:

  • Додати bcrypt замість SHA256;
  • Реалізувати refresh tokens;
  • Додати rate limiting;
  • Покращити error handling;
  • Додати logging та monitoring.

Для десктопу:

  • Auto-update mechanism;
  • Notifications для синхронізації;
  • Tray icon;
  • Keyboard shortcuts.

Для веб:

  • Progressive Web App (PWA);
  • Service Workers для offline;
  • WebSocket для real-time updates;
  • Better mobile UX.

Висновки

Проведено детальний порівняльний аналіз десктоп та веб версій системи управління файлами. Обидві версії мають спільний backend (Django REST API) та більшість frontend коду, що знижує вартість розробки. Десктоп-версія на Electron забезпечує повний функціонал з синхронізацією папок, але вимагає більше ресурсів. Веб-версія надає універсальний доступ з мінімальними вимогами, але обмежена в роботі з файловою системою.

Рекомендований підхід – гібридний: веб-версія як основна для широкої аудиторії, десктоп-версія для power users. Якщо вибирати одну технологію для file manager, пріоритет слід віддати десктопу через критичність функції синхронізації папок.

Обрана архітектура з REST API забезпечує гнучкість, можливість легко додати мобільні додатки та інтеграцію з третіми сервісами. Спільний backend для обох версій знижує вартість підтримки та забезпечує консистентність даних.
